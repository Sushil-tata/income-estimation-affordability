## Layer 9 — Offer / Action Optimization

### Purpose
Translate the frozen Layer G affordability output into executable product decisions:
maximum loan amount by tenor, card line, action code (STP/MANUAL_REVIEW/REFER),

## 9A — Capacity Engine

**Purpose:** Deterministic translation of ADSC into product-level borrowing limits.

**Inputs (read-only from Layer G):**

| Field | Source | Meaning |
|-------|--------|---------|
| `adjusted_income` | Layer G | PolicyIncome (no haircut) |
| `adsc` | Layer G | Available Debt Servicing Capacity = allowable_obligation − existing_obligations |
| `dscr_used` | Layer G | Effective burden ratio cap (min of persona cap, BCI band cap) |
| `existing_obligations` | Layer G | Current monthly debt commitments |

**Outputs (new fields only):**

| Field | Formula | Note |
|-------|---------|------|
| `max_obligation` | `adjusted_income × dscr_used` | Echoed for audit |
| `residual_capacity` | `adsc` | Echoed for audit |
| `max_emi` | `adsc.clip(0)` | Max new monthly obligation |
| `max_loan_{n}m` | `max_emi × annuity_factor(r, n)` | For each tenor n |
| `max_card_line` | `max_emi × card_line_months_equivalent` | Card limit proxy |

**Annuity formula:**
```
PV = EMI × [(1+r)^n − 1] / [r(1+r)^n]
```
where `r = reference_rate_annual / 12`, `n = tenor_months`.

When `adsc ≤ 0`: all outputs = 0 (no capacity to service new debt).

**Parameters [PROVISIONAL]:**
- `reference_rate_annual = 0.18` — internal sizing rate (not customer offer rate)
- `card_line_months_equivalent = 30` — months of EMI capacity → card limit
- `product_tenors_months = [12, 24, 36, 48, 60]`

## 9B — Confidence Engine

**Purpose:** Monotone multiplier on offer sizing based on income estimation reliability.

**Critical design constraint:** BCI is **not** applied here.
BCI already tightened `dscr_used` in Layer G. Applying it again in 9B would
double-penalise — this is prohibited.

**Inputs:**

| Field | Type | Fallback |
|-------|------|---------|
| `persona` | str | Required |
| `p_reliable10` | float [0,1] | Optional — P(income within ±10%) |
| `p_over10` | float [0,1] | Optional — P(income over-estimated by >10%) |
| `model_confidence` | float [0,1] | Fallback proxy |
| `income_interval_width` | float | Fallback proxy |

**Reliability score (primary path — both p signals present):**
```
reliability_score = (w_rel × p_reliable10 + w_over × (1 − p_over10)) / (w_rel + w_over)
```

**Monotonicity proof:**
- ∂score/∂p_reliable10 = w_rel/(w_rel+w_over) > 0 ✓ (higher accuracy → higher score)
- ∂score/∂p_over10 = −w_over/(w_rel+w_over) < 0 ✓ (higher over-estimation → lower score)

**Confidence multiplier:**
```
multiplier = clip(persona_base × reliability_score^α, min_multiplier, persona_base)
```

**Persona base values [PROVISIONAL]:**

| Persona | Base Multiplier | Rationale |
|---------|----------------|-----------|
| PAYROLL | 1.00 | Payslip-backed — maximum confidence |
| L0 | 0.95 | Structured regular credits |
| L1 | 0.85 | Semi-formal income |
| L2 | 0.70 | Volatile/informal — conservative |
| PT | 0.65 | Part-time/gig — high volatility |
| THIN | **0.00** | Policy floor — income infeasible |
| SPARSE | **0.00** | Data-sufficiency failed |

THIN and SPARSE base multipliers are **policy decisions**, not calibration parameters —
they must not change without a governance review.

**Outputs:**

| Field | Meaning |
|-------|---------|
| `reliability_score` | Pre-multiplier diagnostic [0,1] |
| `confidence_multiplier` | Offer-sizing factor [0,1] |
| `confidence_reason_code` | `PRIMARY_RELIABILITY` / `FALLBACK_PROXY` / `ZERO_PERSONA_BASE` |

**Parameters [PROVISIONAL]:**
- `reliability_weight_p_reliable = 0.60`
- `reliability_weight_p_over = 0.40`
- `reliability_alpha = 1.0`
- `min_multiplier = 0.30`

## 9C — Offer Selector

**Purpose:** Generate ranked candidate offers and select the optimal executable action.

**Algorithm:**

1. **Persona routing (hard rules):**
   - THIN / SPARSE → REFER (no override)
   - `adsc ≤ 0` → REFER
   - `confidence_multiplier = 0` → REFER

2. **Candidate generation:** For each eligible tenor n:
   - `offer_amount = max_loan_{n}m × confidence_multiplier`
   - Filter: `offer_amount ≥ min_loan_amount_thb`
   - Filter: back-calculated EMI ≤ `max_emi` (feasibility re-check)

3. **Scoring (fully auditable, no latent model):**
   ```
   score = offer_amount × tenor_preference_weight[n]
   ```

4. **Selection:** Highest score. Tie-break: shorter tenor.

5. **Action code assignment:**

| Persona | Base Action | STP condition |
|---------|------------|---------------|
| PAYROLL, L0, L1 | STP_APPROVE | `max_emi ≥ stp_min_emi_thb` |
| L2, PT | MANUAL_REVIEW | Always |
| THIN, SPARSE | REFER | Always |

6. **Card line:** Included when `persona ∈ {PAYROLL, L0, L1}` and
   `max_card_line × confidence_multiplier ≥ min_card_line_thb`.

**Outputs:**

| Field | Type | Meaning |
|-------|------|---------|
| `action_code` | enum | `STP_APPROVE` / `MANUAL_REVIEW` / `REFER` |
| `verification_intensity` | enum | `NONE` / `LIGHT` / `STANDARD` / `FULL` |
| `offer_amount_recommended` | float | Adjusted offer (post-multiplier) |
| `recommended_tenor_months` | int | Winning tenor |
| `max_offerable_amount` | float | Pre-multiplier loan maximum |
| `include_card_line` | bool | Whether to include card offer |
| `card_line_recommended` | float | Adjusted card line |
| `stp_flag` | bool | True → no human review required |
| `optimization_reason_codes` | list[str] | Full audit trail |

**Parameters [PROVISIONAL]:**
- `min_loan_amount_thb = 10,000`
- `stp_min_emi_thb = 1,000`
- `min_card_line_thb = 5,000`
- `tenor_preference_weights = {12:1.00, 24:1.05, 36:1.05, 48:1.00, 60:0.95}`

## Governance Rules (Layer 9)

1. **Read-only access to upstream:** Layer 9 may only read from `final_output`.
   No upstream field may be modified, overwritten, or shadowed.

2. **confidence_multiplier ≠ income haircut:** The multiplier gates what we *offer*,
   not what the customer can *afford* by policy. `policy_income` is never touched.
   This distinction must appear in every audit log.

3. **THIN / SPARSE are policy floors:** These produce REFER with zero offer.
   This is a governance decision, not a calibration parameter.

4. **Monotonicity must hold:** Higher income reliability → equal or higher offer amount.
   Proven mathematically in 9B and tested in `tests/test_offer_optimizer.py::TestMonotonicity`.

5. **Full audit trail required:** Every offer record must include `optimization_reason_codes`
   explaining the path taken. No black-box decisions.

6. **No expected-value heuristics without calibration data:** The scoring formula
   (`offer_amount × tenor_preference_weight`) is explicitly defined and auditable.
   Do not introduce revenue-optimisation objectives until vintage data is available.

## Calibration Roadmap

### Immediate (Bootstrap phase — current state)
- All parameters set to conservative a-priori estimates
- `p_reliable10` and `p_over10` not yet wired from reliability engine
- ConfidenceEngine falls back to `model_confidence` + `income_interval_width` proxy
- Tests confirm monotonicity and schema correctness

### 3-month milestone
- Wire `p_reliable10` and `p_over10` from reliability model into pipeline
- Switch ConfidenceEngine to `PRIMARY_RELIABILITY` path for all customers
- Collect STP approval + 30-day early-delinquency data

### 6-month vintage
- Calibrate `persona_base_multipliers` vs cohort default rates
- Calibrate `reliability_alpha` for desired conservatism slope
- Validate `stp_min_emi_thb` against STP portfolio performance
- Review `tenor_preference_weights` vs product revenue/risk profile
- Remove [PROVISIONAL] tags from validated parameters

### 12-month vintage
- Full retrain of reliability model (`p_reliable10`, `p_over10`)
- Recalibrate all Layer 9 thresholds with verified outcome data
- Consider monotone-constrained calibration (isotonic regression on confidence curve)
