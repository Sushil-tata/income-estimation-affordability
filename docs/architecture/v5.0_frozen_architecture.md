## Layer A — Data Intake and Observation Window

**Input:** Monthly aggregates only (no transaction-level replay in inference). 2–6 months of data accepted.

**OQS (Observation Quality Score):**

```
oqs_score = w_depth × min(active_months / 6, 1.0)
          + w_density × min(avg_txn_per_month / 10, 1.0)
```

OQS is a continuous model input feature. It enters all downstream models (reliability engine,
BCI data-richness component) as a numeric signal. It is **not** a routing gate or eligibility cutoff.

**Eligibility:** All customers with ≥ 2 months of data enter the pipeline.
SPARSE routing (Layer B) is a downstream classifier decision, not an OQS threshold.

**Feature Validity Matrix (6 groups):**

| Group | Requires | If absent |
|---|---|---|
| G1 — Credit level (3M) | ≥ 2M active | Masked to population median |
| G2 — Credit level (6M) | ≥ 4M active | Masked to population median |
| G3 — Debit / retention (3M) | ≥ 2M active | Masked to population median |
| G4 — Debit / retention (6M) | ≥ 4M active | Masked to population median |
| G5 — Seasonality / CV | ≥ 5M active | Masked to population median |
| G6 — Merchant / MCC | ≥ 3M active + MCC data | Masked to population median |

Masked features are set to population-median (never zero). `feature_masking_count` is always
recorded in the output. Masking count feeds reliability models and BCI data-richness component.

## Layer B — SPARSE Identification

### Definitional Distinction — THIN vs SPARSE

> **THIN** represents customers with low income or minimal account activity who nonetheless have
> usable signal sufficient to produce a policy-floor income estimate.
> THIN cases **are estimable**; they route to a policy floor, not a model.
>
> **SPARSE** represents cases where reliable income estimation is infeasible from available
> information — the data-sufficiency classifier determines that no income figure can be produced
> with acceptable reliability.
> SPARSE cases **are not estimable**; they route to REFER.
>
> These are distinct operational states. **THIN is an income segment. SPARSE is an
> estimation-feasibility judgment.**

### SPARSE Definition

SPARSE is the operational state for cases where a dedicated data-sufficiency classifier predicts
that reliable income estimation is not feasible from available information.

The data-sufficiency classifier is a trained binary model. Its inputs include `oqs_score`,
`active_months`, `feature_masking_count`, and persona signals. Its output is `p_sparse` — a
calibrated probability that the case is data-insufficient.

Cases predicted as SPARSE receive the following fully-specified output record:

```json
{
  "persona": "SPARSE",
  "policy_income": null,
  "income_source": "SPARSE_CLASSIFIER",
  "oqs_score": "<float>",
  "sparse_reason_code": "<string>",
  "feature_masking_count": "<int>",
  "p_reliable10": null,
  "p_over10": null,
  "bci_score": null,
  "bci_band": "VERY_LOW",
  "decision": "REFER",
  "provisional_flags": ["P-03", "P-10"]
}
```

No SPARSE case produces an undefined or partially-populated output. All downstream systems
receive a complete record.

## Layer C — Persona Assignment

**Stage 1:** Binary THIN classifier (gradient boosted). Output: `p_thin`.

**Stage 2 (non-THIN):** Multiclass L0 / L1 / L2 / PAYROLL / PT classifier. Output: probability vector.

**PT assignment:** PT assignment in v5.0 production is a provisional learned scoring model with
a calibrated decision threshold. The decision threshold is an implementation parameter, not an
architectural definition of PT. `[PROVISIONAL — P-01]`

**PAYROLL assignment:** Learned from payroll signal features. Not a rule.

**Persona Stability Smoother (stateless EMA):**

```
smoothed_p(t) = α × p_raw(t) + (1 − α) × smoothed_p(t−1)
```

Switch persona only if: `P(new) − P(current) > δ` in smoothed vector.
α `[PROVISIONAL — P-02]`, δ `[PROVISIONAL — P-02]`.

**Mixture of Experts (MoE):**
When top-class confidence < threshold, blend top-2 persona model predictions weighted by their
probabilities. THIN and PT bypass MoE entirely.
MoE confidence threshold `[PROVISIONAL — P-02]`.

## Layer D — Income Estimation (by Persona)

### PAYROLL
- **PolicyIncome:** Median of verified payroll credits (direct observation).
- **P50:** PolicyIncome (point estimate from observation).
- **Lower-bound interval:** Derived from calibrated historical residual distributions, or omitted
  if not required operationally.
- CQR bypassed. Income is directly observed.
- `income_source = "PAYROLL_OBSERVED"`

### L0 — Stable structured non-payroll
- **Training target:** Median monthly net credit (6M window). Winsorized at 2.5% tails.
- **Model:** Quantile LightGBM producing P35 and P50. `[PROVISIONAL: quantile levels — P-04]`
- **PolicyIncome:** P50 output.
- **CQR calibration:** Persona-stratified.
  Fallback hierarchy: PERSONA (N≥100) → SHRINKAGE (N=30–99) → POOLED (N<30) → NO_CALIBRATION (total<50).
  `[PROVISIONAL: N thresholds — P-05]`
- `income_source = "L0_QUANTILE_MODEL"`

### L1 — Structured income with irregular components
- **Training target:** Retained inflow proxy — an intermediate construct calibrated against
  verified income. The proxy scalar and quantile level are calibration parameters, not frozen.
  `[PROVISIONAL — P-06]`
- **Model:** Quantile LightGBM producing P35 and P50. `[PROVISIONAL: quantile levels — P-04]`
- **PolicyIncome:** P50 output.
- **CQR calibration:** Persona-stratified, same fallback hierarchy as L0.
- `income_source = "L1_QUANTILE_MODEL"`

### L2 — Informal / volatile income
- **Training target:** P25 of net credit distribution (conservative lower bound).
- **Model:** Quantile LightGBM. `[PROVISIONAL: quantile level — P-04]`
- **PolicyIncome:** Model output (already a conservative lower bound — no additional haircut stacked).
- **CQR calibration:** Persona-stratified; L2 typically has highest conformal scores.
- `income_source = "L2_QUANTILE_MODEL"`

### THIN
- **PolicyIncome:** Policy floor (BOT minimum income threshold). Not ML-trained.
- No quantile model. No CQR.
- `income_source = "THIN_POLICY_FLOOR"`

### PT — Pass-Through `[PROVISIONAL — P-07]`
- **v5.0 bootstrap formula:**
  ```
  usable_income_PT = median_credit_6m × min(retention_ratio_6m, retention_ratio_3m)
  ```
  Capped at median recurring credit (salary-like ceiling). Floored at policy minimum.
- **Lower-bound interval:** Derived from calibrated historical residual distributions, or omitted
  if not required operationally.
- **Target state:** Learned PT income model (replaces formula when training data is sufficient).
- `income_source = "PT_FORMULA"` (bootstrap) / `"PT_MODEL"` (target state)
- CQR bypassed in bootstrap phase.

## Layer E — Reliability Engine

**Purpose:** Quantifies estimation risk on the output. Separate from income estimation.

**Two calibrated model outputs per customer:**

| Output | Definition |
|---|---|
| `p_reliable10` | P(|error| ≤ 10%) — probability that PolicyIncome is within 10% of true income |
| `p_over10` | P(PolicyIncome > TrueIncome × 1.10) — probability of material overestimation |

**Error Tolerance Bands:**

| Band | Range | Meaning |
|---|---|---|
| VL | ≤ 5% | Very low error |
| L | 5–10% | Low error — p_reliable10 threshold |
| M | 10–15% | Moderate |
| H | 15–20% | High |
| VH | > 20% | Very high |

**Model inputs:** `oqs_score`, `feature_masking_count`, persona, income band, CQR residuals,
conformal score. OQS is an input feature — it does not replace the reliability model.

THIN and SPARSE produce `p_reliable10 = null`, `p_over10 = null`.

## Layer F — BCI (Business Confidence Index) `[PROVISIONAL — P-08]`

**Purpose:** Composite confidence score summarising estimation quality.
Drives Burden Ratio cap tightening and decisioning.
**BCI never modifies PolicyIncome.**

**Components and weights `[PROVISIONAL — P-08]`:**

| Component | Weight |
|---|---|
| Income stability | 0.30 |
| Data richness (OQS + masking) | 0.20 |
| Persona clarity (top-class probability) | 0.20 |
| Model uncertainty (1 − p_reliable10) | 0.20 |
| Behavioral consistency | 0.10 |

**BCI Bands:**

| Band | Score Range | Burden Ratio Cap | Policy |
|---|---|---|---|
| HIGH | 80–100 | 0.45 | STP |
| MEDIUM | 60–79 | 0.40 | STP |
| LOW | 40–59 | 0.35 | MANUAL_REVIEW |
| VERY_LOW | 0–39 | 0.30 | DECLINE_OR_REFER |

**Persona BCI Ceilings `[PROVISIONAL — P-09]`:**

| Persona | BCI Ceiling | Rationale |
|---|---|---|
| PAYROLL | 100 | Directly observed income — full confidence possible |
| L0 | 88 | Stable structured; high but not certain |
| L1 | 75 | Structured with irregular components |
| L2 | 60 | Volatile / informal — bounded confidence |
| PT | 50 | Bootstrap formula only — moderate confidence ceiling |
| THIN | 30 | Minimal signal; policy floor only |

**AdjustedIncome = PolicyIncome.** No BCI haircut applied to income.
The `income_haircut` field is retained in output for audit and explainability only (always 1.0).

## Layer G — Affordability Engine

**Burden Ratio:**
```
Burden Ratio = Obligations / PolicyIncome
```

**Effective Burden Ratio cap:**
```
burden_cap_used = min(persona_burden_cap, bci_band_burden_cap)
```

**Persona Burden Ratio Caps (monotone in uncertainty):**

| Persona | Cap | Rationale |
|---|---|---|
| PAYROLL | 0.45 | Highest certainty — largest allowable burden |
| L0 | 0.40 | Stable; standard cap |
| L1 | 0.37 | Irregular income — slight tightening |
| L2 | 0.33 | Volatile — conservative cap |
| PT | 0.33 | Bootstrap formula — same floor as L2 |
| THIN | 0.35 | Policy floor; not ML-driven |

Monotonicity invariant: PAYROLL ≥ L0 ≥ L1 ≥ L2 ≈ PT.
A customer with higher income uncertainty cannot receive a higher Burden Ratio allowance than
a more certain segment.

**Max eligible obligation:**
```
max_eligible_obligation = PolicyIncome × burden_cap_used
```

## Decision Matrix

`[PROVISIONAL: threshold values — P-10]`

> **Note:** The values 0.70, 0.20, and 0.35 are policy-operating thresholds, not statistical
> constants. They are subject to calibration against observed income verification outcomes and
> portfolio performance data.

| BCI Band | p_reliable10 ≥ 0.70 | p_over10 ≤ 0.20 | Burden Ratio ≤ burden_cap_used | Decision |
|---|---|---|---|---|
| HIGH | Yes | Yes | Yes | STP_APPROVE |
| HIGH / MEDIUM | Yes | Yes | No | APPROVE_WITH_CHECKS |
| MEDIUM | Partial | Partial | Yes | APPROVE_WITH_CHECKS |
| LOW | Any | Any | Any | REFER_FOR_INCOME_VERIFICATION |
| VERY_LOW | Any | Any | Any | DECLINE |
| SPARSE | — | — | — | REFER |
| THIN | — | — | — | REFER |

No undefined states. Every customer receives exactly one of:
STP_APPROVE | APPROVE_WITH_CHECKS | REFER_FOR_INCOME_VERIFICATION | DECLINE.

## Forbidden in v5.0

The following are explicitly excluded and must not be re-introduced without MRC approval:

- Any `_12m` feature
- BCI income haircut applied to PolicyIncome
- Hard-coded SPARSE threshold on OQS (e.g. OQS < 0.25 = SPARSE)
- Fixed P10 multipliers (e.g. P10 = 0.90 × PolicyIncome)
- PT detection as a rule based on a fixed credit ratio threshold stated as architecture
- Double-stacking of conservative quantile + conformal interval + BCI haircut
- "DSCR" terminology in audit outputs (Burden Ratio is used throughout)
- AutoGluon in production (excluded pending GDZ policy review)
